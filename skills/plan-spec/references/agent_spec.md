# Code Agent Spec

1. 处理 gitee issue 读取、评论、创建任务 issue、更新状态时，依赖已配置且可用的 Gitee MCP。
2. 执行 gitee 相关操作前，应优先使用 Gitee MCP 工具；如果当前环境未配置、无权限或调用失败，应停止对应的 gitee 同步步骤，并向用户说明缺失的能力和需要补齐的信息。
3. 不要猜测仓库 owner、repo、issue number 或 issue 类型；这些信息应来自用户输入、当前上下文或 Gitee MCP 查询结果。
4. 若无法通过 Gitee MCP 确认 issue 类型为“需求”，不得自动进入需求拆解创建任务流程。
5. 判定“remote 指向 gitee”时，应基于 git remote URL（如 origin）包含 gitee.com 或等效 gitee 域名，不得凭经验假定。
6. 当项目 git remote 指向 gitee 且当前任务未指定对应 gitee issue 时，应先通过 gitee MCP 自动创建“需求” issue，再基于该需求拆解并创建“任务” issue。
7. 在自动创建场景中，需求与任务 issue 必须写明关联关系（来源需求、任务归属、验收标准、依赖关系）。
8. 在自动创建需求/任务 issue 前，应先通过 gitee MCP 进行查重；若已存在可复用 issue，则优先复用并建立关联，避免重复创建。
9. 如果需求来自 gitee issue，且 issue 类型为“需求”，先通过 gitee MCP 读取 issue 内容并进行需求分析，不直接进入实现。
10. 需求分析应覆盖目标、用户价值、范围边界、约束、验收标准、风险和依赖。
11. 将 gitee 需求 issue 拆解为一个或多个可独立执行、可验证的任务，并通过 gitee MCP 创建对应任务 issue。
12. 每个任务 issue 的正文应包含来源需求 issue、任务目标、实现范围、验收标准、测试建议和依赖关系。
13. 在原需求 issue 下评论任务拆解结果，列出创建的任务 issue，并停止实现流程；只有用户明确指定某个任务 issue 为执行对象时，才继续执行。
14. 每次计划模式生成计划后，在执行前先把计划保存到 `spec` 目录下，命名格式统一为：`spec/feats/序号-简要描述-日期.md`。其中序号为三位递增数字，简要描述为英文缩写，日期格式为 `YYYYMMDD`，例如 `spec/feats/003-ticket-order-pay-20241203.md`。
15. 计划文件结构标题为功能的中文摘要，例如 `## 门票订单支付流程优化`。副标题为当前的日期和时间。正文为本次计划的详细内容，包括变更点、测试计划、假设等。
16. 计划执行前，再看下之前的计划文件，确认下功能是否有冲突或者重复，如果有，先评估下是否需要调整计划内容或者合并计划。如不能确定先问我明确后再执行，先不要直接执行计划。
17. 本次计划执行完成后，更新计划文件，增加执行结果的总结和后续跟进事项等内容，保持计划文件的完整性和可追溯性。
18. 如果功能特性关联到 gitee 上的任务 issue，完成后上传计划文件相关内容到 issue 中并按场景更新状态：常规流程更新为“已提测”；自动创建场景需记录任务开始时间和结束时间，并在完成后将任务 issue 与对应需求 issue 更新为 closed。
